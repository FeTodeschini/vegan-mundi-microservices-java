Architecture Guide

Vegan Mundi Java Microservices on AWS

This document explains the current Java migration architecture in plain language: what each service does, how traffic flows through AWS, how Jenkins delivers changes, and how Terraform organizes the infrastructure.

At a glance

  • 7 Spring Boot services plus 1 shared library and 1 Lambda module
  • AWS ECS on EC2 behind an Application Load Balancer
  • Terraform modules for network, compute, registry, logging, events, and IAM
  • Jenkins pipeline for build, image publish, infrastructure rollout, and smoke tests
System shape

How the platform is put together

The architecture uses a classic edge-to-core layout. Client traffic reaches an Application Load Balancer, which routes requests to Java services running as containers on ECS. Services use a shared library for common concerns such as JWT handling, and the order path also publishes an asynchronous event for a Lambda-based follow-up workflow.

Clients Web browsers, mobile clients, or API callers.
DNS External DNS points traffic to the ALB endpoint.
ALB Path-based routing sends traffic to the right service.
ECS Services Spring Boot containers handle the request.
DOWNSTREAM FLOW
MySQL Primary relational data store for service-owned tables.
EventBridge Receives order domain events for async processing.
Lambda Handles order confirmation and optional analytics work.
CloudWatch Captures logs, metrics, alarms, and service visibility.
Component map

Microservices and supporting components

account-service

Owns registration, login, profile lookup, and JWT-oriented authentication concerns.

Expected routes include /api/account/register, /login, /profile, and /health.

class-service

Provides class catalog search, filters, lookups, and category-driven browsing.

Supports the product discovery side of the platform.

order-service

Coordinates checkout, order creation, and order status updates.

This is the most integration-heavy service because it touches pricing, delivery, and async events.

review-service

Manages user reviews, star ratings, and feedback attached to classes.

Read-heavy and largely independent from order processing.

delivery-service

Models delivery methods, shipment details, and shipping-related calculations.

Used by the order path to calculate or validate fulfillment options.

gallery-service

Handles media metadata and S3-oriented image or video workflows.

Useful for media hosting and asset management without bloating other services.

price-service

Owns pricing rules, discounts, and cost estimation logic.

Designed to keep pricing logic isolated from order orchestration.

shared-library and Lambda

The shared library centralizes common code. The order-confirmation Lambda handles async side effects after an order event.

This split keeps synchronous requests lean while still supporting event-driven behavior.
Request paths

How data moves through the system

Synchronous request flow

  • A user request reaches the ALB.
  • The ALB chooses the target service using the URL path.
  • The service executes business logic and reads or writes MySQL.
  • If the workflow needs another service, ECS-based service discovery is the intended internal path.
  • The service returns a normal HTTP response to the client.

Asynchronous order flow

  • order-service completes the core checkout transaction.
  • It publishes an order event to EventBridge.
  • The order-confirmation Lambda reacts after the client response is already done.
  • The Lambda can send confirmation email with SES and record lightweight analytics data.
  • This keeps customer-facing latency separate from follow-up processing.
Delivery automation

What the Jenkins pipeline is responsible for

The repo includes a declarative Jenkins pipeline plus a dedicated Jenkins runtime folder. The design assumes Jenkins can build the whole Maven project, publish Docker images to ECR, optionally run Terraform, and then trigger ECS deployments.

1

Validate and checkout

The pipeline validates required parameters such as AWS account ID and Terraform state bucket, then checks out the repo.

2

Build and test

Maven builds all modules. Tests can be skipped with a parameter, but the intended default path runs them.

3

Build and push container images

Each service image is built and pushed to Amazon ECR with a build-number tag plus latest.

4

Plan infrastructure changes

If enabled, Jenkins runs terraform init and terraform plan inside the chosen environment folder and writes a readable plan artifact.

5

Approval gate and apply

A manual approval step protects the apply stage. If approved and not in dry-run mode, Terraform applies the saved plan.

6

Deploy ECS services

The pipeline reads current task definitions, swaps container image references, registers new task definitions, and forces new ECS deployments.

7

Smoke test and rollback

Health endpoints are checked through the ALB. On failure, the pipeline hands off to a rollback script.

Infrastructure as code

How Terraform is organized

Terraform is laid out in the standard reusable-module style. The root defines shared provider and backend behavior, while environment folders choose concrete values and instantiate the module set.

Root layer

  • backend.tf configures remote state in S3.
  • variables.tf defines region, capacity, database, network, and certificate controls.
  • outputs.tf is intended to publish ALB, ECS, registry, database, and event outputs.

Environment layer

  • terraform/dev holds one environment instance and also contains evidence of a past apply.
  • terraform/test and terraform/prod mirror the same pattern for higher environments.
  • terraform/jenkins manages a dedicated Jenkins host stack.

Module layer

  • vpc for network and subnet layout.
  • alb for public ingress and target groups.
  • ecs for cluster, launch template, ASG, and capacity provider.
  • ecr, iam, rds, cloudwatch, eventbridge, and lambda for the rest of the platform.
Operational reading

What this architecture says about the project

Strengths

  • Clear separation of service concerns.
  • Infrastructure and delivery are described in code, not in tribal knowledge.
  • The design demonstrates both container-based and event-driven patterns.
  • The repo is structured so a single service can be proved first and the rest can follow.

Current reality

  • This is a strong foundation, not a finished production platform.
  • Several service endpoints are still stubs and need real persistence and security logic.
  • Terraform and Jenkins both need environment validation to prove that the coded design matches live behavior.
  • Some Markdown docs referenced elsewhere in the repo are still missing and should be filled in as Phase 1 proceeds.
